## 🎯 一屏精炼卡片

> 目标：不看正文，只看这张卡，30 秒抓住全文。卡片必须能独立成立（发朋友圈/群里也读得懂）。

> **TL;DR（一句话）**：<问题 + 核心做法 + 最关键结果，尽量 40 字内>

| 维度 | 内容 |
|---|---|
| 🧩 **要解决的问题** | 一句话：论文瞄准的真实痛点是什么 |
| 💡 **核心洞见 / 最关键的一招** | 如果全文只记一句话，就是这句（方法背后的 key idea / trick） |
| 🛠️ **方法一句话** | 怎么做到的（pipeline 或关键机制） |
| 📊 **关键结果** | 最有说服力的 1–2 个数字（相对 baseline 的提升） |
| ⚖️ **何时该用 / 不该用** | 适用场景与已知失效场景 |
| 🖼️ **一图胜千言** | 指向最能代表全文的那张图（如 `Fig. 2` 架构图），并一句话说它为什么关键 |
| 🏷️ **关键词** | 3–5 个 |
| ⭐ **精读者评级** | 必读 / 值得读 / 速览 —— 附一句理由 |

## WHAT · 作者做了什么

### 2.1 核心任务

- 论文要解决的具体任务/问题是什么？
- 输入、输出、评估场景分别是什么？
- 如果有任务定义图，嵌入在这里并解释：

  ```markdown
  ![Figure 1: Task illustration](figures/figure-001-task.png)
  ```

  **图注解读**：这张图展示了 ...（按从左到右 / 从上到下的顺序解释）。它说明 paper 把 xxx 定义为 ...（Evidence: Sec. 2, Fig. 1）。

### 2.2 主要贡献

逐条列出作者自己声明的贡献，每条用 `[C2.2.x]` 标注并附带证据：

- **[C2.2.1]** 贡献 1：...（Evidence: Abstract, Sec. 1）
- **[C2.2.2]** 贡献 2：...（Evidence: Sec. 5）

### 2.3 与已有工作的关键差异

- 最接近的相关工作是什么？
- 这篇论文与它们的核心区别在哪里？（方法/设置/数据/发现）
- 如果有对比图/表，嵌入在这里。

## WHY · 作者为什么要做这件事

### 3.1 背景问题

- 这个任务所属的大方向是什么？
- 在工业界或学术界，这个问题为什么存在？

### 3.2 现有方法不足

- 现有方法通常怎么做？
- 它们的瓶颈、假设或失败场景是什么？
- 如果有 gap/illustration 图，嵌入在这里。

### 3.3 作者动机

- 作者想打破的假设或想填补的 gap 是什么？
- 这个 gap 是作者直接指出的，还是你从多处证据推断的？

### 3.4 问题重要性

- 这个问题解决后会影响哪些下游任务或应用？
- 如果不解决，会导致什么实际代价？

## HOW · 作者具体怎么做

### 4.1 方法总览

- 用 3-5 句话描述整体 pipeline。
- 如果有整体架构图，嵌入在这里：

  ```markdown
  ![Figure 2: Overall framework](figures/figure-002-framework.png)
  ```

  **图注解读**：输入是 ...，经过模块 A/B/C ...，输出 ...。箭头 ... 表示 ...（Evidence: Sec. 3.1, Fig. 2）。

### 4.2 流程逐步拆解

按时间/数据流顺序，把 pipeline 拆成 4-7 步。每步包含：

1. **Step 1**：做什么、输入、输出、为什么需要这步。
2. **Step 2**：...

### 4.3 模块级细读

对每个关键模块使用下面的结构。如果模块很多，可以用子节 `#### 模块 X：名称`。

```markdown
#### 模块：<名称>

- **目的**：这个模块解决什么问题？
- **输入**：
- **输出**：
- **内部步骤**：
- **相关公式或配图**：
- **设计理由**：为什么这样设计，而不是更直接的替代方案？
- **证据**：Sec. X, Eq. Y, Fig. Z
```

### 4.4 公式与关键技术细节

对每一个关键公式：

```markdown
#### 公式 <n>：<名称>

原始公式：
$$
...
$$

| 符号 | 含义 | 定义位置 |
|---|---|---|
| $x$ | ... | Sec. 3.2 |

- **通俗解释**：...
- **在方法流程中的作用**：...
- **直观理解 / toy example**：...
- **证据**：...
```

### 4.5 训练 / 优化 / 推理细节

- 损失函数、优化器、学习率、batch size 等关键超参数。
- 推理时的特殊设置（beam search、采样温度、并行策略等）。
- 实现层面的重要取舍。

### 4.6 实现细节

- 代码框架、语言、大概规模。
- 公开代码中暴露的关键类/函数。
- 论文中没有明说但实现中需要注意的点。

## 10. 认知启示与应用拓展（Enlightened · 编者视角）

> ⚠️ **本节规则**：以上章节忠实转述论文；本节是**精读者的推断与延展**，允许超出论文原文，但每条尽量标注这是「推断」而非「论文声称」。这一节是本报告区别于「论文摘要」的核心——目标是**读完能生出可行动的新认知**，而不只是复述作者做了什么。禁止空话套话，每条都要具体到「能落地/能验证/能反驳」。

### 10.1 最反直觉 / 最值得记住的一点

- 读完全文，最颠覆你原有认知、或最想记住的一句判断是什么？为什么它反直觉？

### 10.2 技术迁移：这套思路还能用在哪

- 把方法**拆成可复用的零件**：哪个模块 / trick / 表示 / 训练技巧可以单独拿走用到别的问题上？
- 迁移到某个具体的其他任务（点名 X）会怎样？需要改哪里、可能在哪失效？
- 与你熟悉的某个方法组合会发生什么？（1+1 的可能性）

### 10.3 业界应用与商业价值

- 哪些**真实产品 / 场景**能直接受益？谁会为它买单？
- 距离生产可用还差什么？（数据规模、算力/成本、延迟、可靠性、合规）
- 如果要做成一个 demo / 内部工具，最小可行版本长什么样？

### 10.4 二阶思考：如果这个方向成立

- 顺着这条路走下去，1–2 年后这个子领域会发生什么变化？
- 它对**我 / 我的团队**当前的工作意味着什么——该立刻跟进、纳入技术选型，还是持续观望？给出明确建议。
- 它可能让什么现有方案/产品**过时**？

### 10.5 可延伸的研究 / 工程 idea

用 `[I<n>]` 编号，每条给「想法 + 为什么可能成立 + 如何最快验证」：

- **[I1]** ...
- **[I2]** ...
- **[I3]** ...

### 10.6 一句话认知升级

- 「这篇论文让我对 ___ 的理解，从 ___ 变成了 ___。」
